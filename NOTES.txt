--------------------------------------------------------------------------------
WHAT THIS PROJECT IS
--------------------------------------------------------------------------------

WinImageTool is a custom Windows image customization and debloating tool built
in C# (.NET 9 / WPF). It is inspired by NTLite but fully open and extensible.
It lets you take a raw Windows ISO/WIM file, strip out bloatware, apply
performance tweaks, integrate drivers and updates, configure unattended setup,
and produce a clean bootable ISO - all from a single GUI or CLI.

The project was started because:
  - NTLite's free tier locks out the most useful features (component removal,
    live image editing, driver exclusion, etc.)
  - We wanted something fully controllable and extendable
  - Several high-quality MIT-licensed open source projects exist that we can
    legally borrow from and build on

--------------------------------------------------------------------------------
KEY TECHNICAL DECISIONS
--------------------------------------------------------------------------------

1. WHY C# + .NET 9?
   - Native Windows API access (DISM, registry, P/Invoke for wimlib)
   - WPF for rich native GUI (same framework as most serious Windows tools)
   - Same binary runs as both GUI (WPF) and CLI (console)
   - Strong typing makes the complex registry and DISM logic safe

2. WHY THREE PROJECTS (Core, GUI, CLI)?
   - Core has zero UI dependencies - can be tested, reused, or called from
     PowerShell scripts
   - GUI and CLI both reference Core and add their own UX on top
   - Keeps concerns clean: no WPF in Core, no DISM in ViewModels

3. DISM API VS DISM.EXE
   - Microsoft.Dism NuGet is used for most operations (mount, packages,
     features, drivers, updates) - it gives typed results and progress callbacks
   - For AppX provisioned package removal we call dism.exe directly because
     DismApi.RemoveProvisionedAppxPackage exists but returns collection types
     that are easier to query via the raw text output approach tiny11 uses
   - For registry hive loading we use reg.exe (same as tiny11builder) because
     Windows doesn't expose a public managed API for offline hive loading

4. OFFLINE REGISTRY HIVE PATTERN
   - Mount the image with DISM to a scratch directory
   - Load hives: reg load HKLM\zSOFTWARE <mountpath>\Windows\System32\config\SOFTWARE
   - Apply tweaks directly to HKLM\zSOFTWARE\... paths via Registry API
   - Unload: reg unload HKLM\zSOFTWARE
   - Unmount the image with commit=true to save
   - This is exactly how tiny11builder does it (proven approach)

5. WIMLIB USAGE
   - wimlib is LGPL - we can use the DLL as-is without open-sourcing our code
   - It gives us WIM/ESD metadata reading that DISM's managed API doesn't
     expose as cleanly (image names, sizes, versions from XML)
   - We load it with NativeLibrary.Load() + delegate-based P/Invoke
   - The DLL path is resolved relative to AppContext.BaseDirectory\tools\wimlib\x64\

6. ISO BUILDING
   - Uses oscdimg.exe from the Windows ADK
   - Dual-boot command: -bootdata:2#p0,e,b<etfsboot.com>#pEF,e,b<efisys.bin>
   - Falls back gracefully if oscdimg is not found (IsoBuilder.IsOscdimgAvailable())
   - Windows ADK is free to install from Microsoft

HOW TO BUILD AND RUN
--------------------------------------------------------------------------------

Requirements:
  - Windows 10/11
  - .NET 9 SDK (installed via winget: Microsoft.DotNet.SDK.9)
  - Optional: Windows ADK (for ISO building with oscdimg)
  - Admin rights at runtime (DISM and registry operations require elevation)

Build:
  cd E:\Projects\WinImageTool
  dotnet build WinImageTool.sln

Run GUI:
  dotnet run --project src\WinImageTool.GUI\WinImageTool.GUI.csproj

Run CLI:
  dotnet run --project src\WinImageTool.CLI\WinImageTool.CLI.csproj -- --help
  dotnet run --project src\WinImageTool.CLI\WinImageTool.CLI.csproj -- info <path-to.wim>
  dotnet run --project src\WinImageTool.CLI\WinImageTool.CLI.csproj -- tweak --list
  dotnet run --project src\WinImageTool.CLI\WinImageTool.CLI.csproj -- unattended -o autounattend.xml
